# Maelstrom 文档组织规范

## 目录结构

```
docs/
├── README.md                ← 本文件
├── start/                   ← 架构愿景与系统设计（编号 01-11）
├── V0/                      ← V0 版本实现文档
│   ├── P0-*.md              ← P0 批次：核心骨架 & QA Chat
│   ├── P1-*.md              ← P1 批次：Gap Engine 全链路
│   └── changes/             ← 后续小修小改（见下方规范）
└── V1/                      ← V1 版本（未来）
```

## 命名规范

### 大版本建设阶段（P0 / P1 / ...）

用于集中建设期，每个 doc 对应一个完整的 feature spec。

格式：`P{优先级}-{序号}_{snake_case_名称}.md`

```
P0-00_project_skeleton.md
P1-05_gap_engine_graph.md
```

- 优先级从 0 开始，数字越小越先做
- 序号在同一优先级内自增
- 同一批次内的 doc 有明确的依赖链

### 小修小改阶段（changes/）

P0/P1 批次完成后，进入持续迭代。零散改动放入 `changes/` 目录。

格式：`{类型}-{三位编号}_{snake_case_名称}.md`

```
FIX-001_chat_sse_reconnect.md
ENH-002_dark_mode_toggle.md
```

四种类型前缀：

| 前缀 | 含义 | 场景 |
|------|------|------|
| `FIX` | Bug 修复 | 运行时报错、逻辑错误、UI 异常 |
| `ENH` | 小功能增强 | 不改架构的功能补充、UX 改善 |
| `PERF` | 性能优化 | 缓存、查询优化、减少重渲染 |
| `CHORE` | 工程杂务 | 脚本、CI、依赖升级、文档补充 |

每种类型编号独立自增（FIX-001、FIX-002... 与 ENH-001、ENH-002... 互不干扰）。

## 单篇文档模板

所有 doc（无论 P0 还是 changes）统一使用以下结构，小改动可酌情精简：

```markdown
# {编号}: {标题}

## 依赖
- 前置任务编号，无则写"无"

## 目的
一句话说明为什么要做这件事。

## 执行方法
1. 具体步骤
2. 涉及的文件和改动

## 验收条件
- 可观测的完成标准
```

## 版本切换

当启动新大版本（如 V1）时：

1. 新建 `docs/V1/` 目录
2. 重新从 P0 开始组织批次任务
3. 上一版本的 `changes/` 目录冻结，不再追加
